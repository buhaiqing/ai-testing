## 一、背景与问题定义

### 1.1 为什么需要这个机制？

在已确定的 CQRS + Event Sourcing 架构基础上，我们发现系统还面临以下核心挑战：

| 挑战 | 具体表现 | 现有架构的不足 |
|-----|---------|--------------|
| **阈值僵化** | 置信度阈值固定（如 0.85/0.95），无法适应不同业务场景的风险偏好 | 一刀切的阈值导致高风险场景漏判或低风险场景过度拦截 |
| **Agent 博弈** | 多个专业 Agent 可能为了自身目标而损害全局目标 | 缺乏机制确保 Agent 个体理性与集体理性一致 |
| **激励扭曲** | 如果 Generator Agent 的 KPI 是"生成数量"，可能牺牲质量换取数量 | 单一指标导致行为偏离设计意图 |
| **动态环境** | 系统负载、数据质量、模型能力都在变化，静态规则难以应对 | 需要自适应的调整机制 |

### 1.2 核心问题

> **如何设计一套机制，使得：**
> 1. 系统的决策阈值能够根据环境动态调整
> 2. 每个 Agent 在追求自身利益的同时，自然促进全局目标
> 3. 避免 Agent 之间的策略性行为损害系统整体性能

## 二、动态阈值机制设计

### 2.1 设计原则

| 原则 | 说明 | 来源 |
|-----|------|------|
| **上下文感知** | 阈值应基于当前任务特征、系统状态、历史表现动态计算 | 控制理论 |
| **风险分级** | 不同风险等级的操作应有不同的阈值曲线 | 金融风控实践 |
| **渐进调整** | 阈值变化应平滑，避免剧烈跳变导致系统不稳定 | 反馈控制理论 |
| **可解释性** | 阈值的计算过程应透明，便于人工审计 | 可信 AI 原则 |

### 2.2 动态阈值计算模型

#### 2.2.1 输入变量

```
动态阈值 = f(基础阈值, 风险系数, 环境因子, 历史表现, 实时负载)
```

| 变量 | 说明 | 取值范围 | 数据来源 |
|-----|------|---------|---------|
| **基础阈值** | 系统默认的静态阈值 | [0, 1] | 配置文件 |
| **风险系数** | 当前操作的风险等级 | [0.5, 2.0] | 规则引擎 |
| **环境因子** | 当前系统健康度 | [0.8, 1.2] | 监控指标 |
| **历史表现** | Agent / 任务类型的历史成功率 | [0.5, 1.5] | Event Store |
| **实时负载** | 系统当前并发量 / 队列深度 | [0.8, 1.5] | 系统指标 |

#### 2.2.2 风险分级矩阵

| 操作类型 | 风险等级 | 风险系数 | 示例 |
|---------|---------|---------|------|
| **只读查询** | 极低 | 0.5 | 查看测试报告、查询历史记录 |
| **测试生成** | 低 | 0.8 | 生成单元测试代码 |
| **测试执行** | 中 | 1.0 | 在沙箱中执行测试 |
| **代码提交** | 高 | 1.3 | 将生成的测试合并到主分支 |
| **生产部署** | 极高 | 2.0 | 触发 CI/CD 生产部署 |

#### 2.2.3 动态阈值公式（初版）

```python
def calculate_dynamic_threshold(
    base_threshold: float,      # 基础阈值（如 0.85）
    risk_level: str,            # 风险等级
    system_health: float,       # 系统健康度 [0, 1]
    agent_history: dict,        # Agent 历史表现
    current_load: float,        # 当前负载 [0, 1]
) -> float:
    """
    动态阈值计算
    
    设计思路：
    1. 高风险操作提高阈值（更严格）
    2. 系统不健康时提高阈值（更保守）
    3. Agent 历史表现好可降低阈值（更信任）
    4. 高负载时适当提高阈值（避免错误放大）
    """
    
    # 风险系数映射
    risk_multiplier = {
        "very_low": 0.5,
        "low": 0.8,
        "medium": 1.0,
        "high": 1.3,
        "critical": 2.0,
    }[risk_level]
    
    # 环境因子：系统越不健康，阈值越高
    environment_factor = 1.0 + (1.0 - system_health) * 0.3
    
    # 历史因子：Agent 历史成功率越高，阈值可适当降低
    success_rate = agent_history.get("success_rate", 0.8)
    history_factor = 1.0 - (success_rate - 0.8) * 0.2  # [0.96, 1.04]
    
    # 负载因子：负载越高，阈值越高
    load_factor = 1.0 + max(0, current_load - 0.7) * 0.3
    
    # 综合计算
    dynamic_threshold = base_threshold * risk_multiplier * environment_factor * history_factor * load_factor
    
    # 边界保护
    return min(0.99, max(0.5, dynamic_threshold))
```

#### 2.2.4 阈值调整效果示例

| 场景 | 基础阈值 | 风险系数 | 环境因子 | 历史因子 | 负载因子 | **动态阈值** | 说明 |
|-----|---------|---------|---------|---------|---------|------------|------|
| 常规测试生成 | 0.85 | 0.8 | 1.0 | 1.0 | 1.0 | **0.68** | 低风险，阈值降低 |
| 高风险代码提交 | 0.85 | 1.3 | 1.0 | 1.0 | 1.0 | **1.11** | 超过上限，取 0.99 |
| 系统故障期间 | 0.85 | 1.0 | 1.2 | 1.0 | 1.0 | **1.02** | 超过上限，取 0.99 |
| 高负载+优秀Agent | 0.85 | 1.0 | 1.0 | 0.96 | 1.09 | **0.89** | 综合调整后略升 |
| 新Agent+生产部署 | 0.85 | 2.0 | 1.0 | 1.04 | 1.0 | **1.77** | 超过上限，取 0.99 |

## 三、激励相容机制设计

### 3.1 博弈论基础

#### 3.1.1 核心概念

| 概念 | 定义 | 在质量智能体中的映射 |
|-----|------|-------------------|
| **激励相容 (Incentive Compatibility)** | 设计机制使得参与者如实报告私有信息是最优策略 | Agent 如实报告自身能力和置信度是最优策略 |
| **个体理性 (Individual Rationality)** | 参与者参与机制的效用不低于不参与 | Agent 参与协作的效用高于独立工作 |
| **社会最优 (Social Optimality)** | 机制实现全局效用最大化 | 系统整体测试质量最高、成本最低 |
| **策略证明 (Strategy-proof)** | 参与者无法通过虚报信息获得更高效用 | Agent 无法通过虚报置信度获得更高奖励 |

#### 3.2.2 Agent 效用函数设计

```python
class AgentUtilityCalculator:
    """
    Agent 效用计算器
    
    设计目标：
    1. 鼓励高质量输出
    2. 鼓励诚实报告
    3. 鼓励协作共享
    4. 惩罚搭便车和虚报行为
    """
    
    def calculate_utility(
        self,
        agent_id: str,
        task_result: TaskResult,
        reported_confidence: float,
        actual_quality: float,
        collaboration_score: float,
        resource_usage: float,
    ) -> float:
        """
        计算 Agent 本次任务的效用
        
        参数:
            task_result: 任务执行结果
            reported_confidence: Agent 报告的置信度
            actual_quality: 实际质量评分（由 Critic/Arbiter 评估）
            collaboration_score: 协作贡献分（是否共享知识、协助其他 Agent）
            resource_usage: 资源使用量（CPU、内存、Token）
        """
        
        # 1. 质量贡献（核心正向激励）
        quality_contribution = actual_quality * 100  # [0, 100]
        
        # 2. 诚实奖励（激励相容关键）
        # 报告置信度与实际质量的差异越小，奖励越高
        confidence_error = abs(reported_confidence - actual_quality)
        honesty_bonus = max(0, 20 - confidence_error * 100)  # [0, 20]
        
        # 3. 协作奖励
        collaboration_bonus = collaboration_score * 10  # [0, 10]
        
        # 4. 资源效率（鼓励节约）
        expected_usage = task_result.complexity * 10  # 预期资源
        efficiency = min(1.0, expected_usage / max(resource_usage, 1))
        efficiency_bonus = efficiency * 10  # [0, 10]
        
        # 5. 虚报惩罚（关键负向激励）
        # 如果报告置信度远高于实际质量，严厉惩罚
        if reported_confidence > actual_quality + 0.1:
            misreport_penalty = (reported_confidence - actual_quality) * 200
        else:
            misreport_penalty = 0
        
        # 6. 故障成本
        if task_result.status == "failed":
            failure_cost = 50
            # 如果虚报导致故障，额外惩罚
            if reported_confidence > 0.9:
                failure_cost += 100
        else:
            failure_cost = 0
        
        # 综合效用
        utility = (
            quality_contribution +
            honesty_bonus +
            collaboration_bonus +
            efficiency_bonus -
            misreport_penalty -
            failure_cost
        )
        
        return max(-100, utility)  # 下限保护
```

#### 3.2.3 信誉系统

```python
class ReputationSystem:
    """
    Agent 信誉系统
    
    设计原则：
    1. 信誉积累慢，下降快（保护系统免受突然的信任攻击）
    2. 历史表现权重指数衰减（更重视近期表现）
    3. 信誉影响资源分配和决策权重
    """
    
    def __init__(self):
        self.reputations = {}  # agent_id -> ReputationRecord
        self.decay_factor = 0.95  # 历史效用衰减因子
    
    def update_reputation(
        self,
        agent_id: str,
        task_utility: float,
        task_difficulty: float,
    ):
        """
        更新 Agent 信誉
        
        更新规则：
        - 高难任务的高效用带来更大信誉提升
        - 低信誉 Agent 需要更多正面表现才能恢复
        """
        record = self.reputations.get(agent_id, ReputationRecord(
            agent_id=agent_id,
            score=100.0,
            history=[],
        ))
        
        # 信誉更新公式
        # 新信誉 = 旧信誉 * 衰减 + 本次效用 * 难度系数 * 学习率
        learning_rate = 0.1
        difficulty_multiplier = 1.0 + (task_difficulty - 0.5)  # [0.5, 1.5]
        
        new_score = (
            record.score * self.decay_factor +
            task_utility * difficulty_multiplier * learning_rate
        )
        
        # 边界保护
        record.score = min(200, max(0, new_score))
        record.history.append((datetime.utcnow(), task_utility, new_score))
        
        self.reputations[agent_id] = record
    
    def get_reputation_weight(self, agent_id: str) -> float:
        """
        获取 Agent 的信誉权重
        
        用途：
        1. 共识投票时的权重
        2. 资源分配时的优先级
        3. 动态阈值调整时的信任系数
        """
        score = self.reputations.get(agent_id, ReputationRecord()).score
        # 将 [0, 200] 映射到 [0.5, 1.5]
        return 0.5 + score / 200
```

### 3.3 VCG 资源分配机制

#### 3.3.1 问题定义

多个 Agent 竞争有限资源（CPU、内存、LLM Token 配额），如何设计分配机制使得 Agent 如实报告真实需求？

#### 3.3.2 VCG 机制原理

**Vickrey-Clarke-Groves (VCG)** 机制是机制设计中的经典方案，核心特性：
- **真实报价是最优策略**（策略证明性）
- **社会效用最大化**（分配效率）

#### 3.3.3 在质量智能体中的应用

```python
class VCGResourceAllocator:
    """
    基于 VCG 机制的资源分配器
    
    应用场景：
    - LLM Token 预算分配
    - 沙箱执行资源分配
    - 任务并行度分配
    """
    
    def allocate(
        self,
        agents: List[Agent],
        total_resources: float,
    ) -> Dict[str, float]:
        """
        VCG 资源分配
        
        步骤：
        1. 收集各 Agent 的真实效用函数（通过信誉系统校准）
        2. 计算社会最优分配（最大化总效用）
        3. 计算每个 Agent 的 VCG 支付（外部性成本）
        """
        
        # Step 1: 收集报价（Agent 报告其对资源的估值）
        bids = {}
        for agent in agents:
            bids[agent.id] = agent.report_value(total_resources / len(agents))
        
        # Step 2: 社会最优分配（贪心或线性规划）
        allocation = self._social_optimal_allocation(bids, total_resources)
        
        # Step 3: 计算 VCG 支付
        payments = {}
        for agent in agents:
            # 计算该 Agent 不参与时的社会最优总效用
            other_agents = [a for a in agents if a.id != agent.id]
            other_bids = {k: v for k, v in bids.items() if k != agent.id}
            other_total = sum(self._social_optimal_allocation(other_bids, total_resources).values())
            
            # 该 Agent 参与时其他 Agent 的总效用
            with_agent_total = sum(
                allocation[a.id] for a in other_agents
            )
            
            # VCG 支付 = 外部性成本
            payments[agent.id] = other_total - with_agent_total
        
        return {
            "allocation": allocation,  # 分配结果
            "payments": payments,       # VCG 支付（可设计为信誉调整而非货币）
        }
```
