## 一、问题背景与必要性

### 1.1 为什么需要零信任 Agent 网格

在去中心化共识架构（Raft）中，虽然消除了单点故障，但引入了新的安全挑战：

| 新风险 | 说明 | 潜在影响 |
|-------|------|---------|
| **恶意 Agent 注入** | 攻击者伪装成合法 Agent 加入共识网络 | 篡改共识日志、伪造任务结果 |
| **横向移动** | 一个被攻破的 Agent 可以访问其他 Agent | 权限扩散、数据泄露 |
| **通信窃听** | Agent 间明文通信被截获 | 敏感测试数据、代码泄露 |
| **权限滥用** | Agent 拥有超出其职责的权限 | 越权操作、数据破坏 |
| **供应链攻击** | 第三方 Agent 组件被植入后门 | 整个系统被控制 |

### 1.2 与传统零信任架构的区别

| 维度 | 传统零信任（IT 系统） | 零信任 Agent 网格（AI 系统） |
|------|---------------------|---------------------------|
| **信任主体** | 用户、设备 | Agent（AI 实体） |
| **验证对象** | 身份、设备健康 | Agent 身份、行为模式、输出可信度 |
| **威胁模型** | 外部攻击者 | 外部攻击者 + 恶意/失控 Agent |
| **动态性** | 相对静态 | Agent 行为高度动态，需实时评估 |
| **可解释性** | 明确 | Agent 决策黑盒，需额外验证机制 |

## 二、核心设计原则

### 2.1 五大安全原则

```
┌─────────────────────────────────────────────────────────────────┐
│                 零信任 Agent 网格五大原则                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 永不信任，始终验证 (Never Trust, Always Verify)              │
│     → 每个 Agent 的每次请求都必须验证身份和权限                      │
│                                                                 │
│  2. 最小权限原则 (Least Privilege)                               │
│     → Agent 只能访问完成任务所需的最小资源集合                       │
│                                                                 │
│  3. 假设已被攻破 (Assume Breach)                                 │
│     → 设计时假设某个 Agent 已被控制，限制其影响范围                   │
│                                                                 │
│  4. 动态风险评估 (Dynamic Risk Assessment)                        │
│     → 基于 Agent 行为实时调整信任等级和权限                         │
│                                                                 │
│  5. 全链路可观测 (Full Observability)                            │
│     → 所有 Agent 行为、通信、决策必须可审计、可追溯                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 与现有架构的集成点

| 现有组件 | 零信任增强 | 实现方式 |
|---------|-----------|---------|
| **Raft 共识层** | 节点身份验证 + 通信加密 | mTLS + SPIFFE/SPIRE |
| **Event Bus** | 事件签名验证 + 防篡改 | 数字签名 + 哈希链 |
| **Agent 注册** | 自动身份颁发 + 定期轮换 | SPIFFE/SPIRE 工作负载身份 |
| **共享记忆层** | 字段级加密 + 访问审计 | AES-256-GCM + 审计日志 |
| **工具层** | 命令注入防护 + 沙箱隔离 | gVisor + seccomp |

### 3.2 身份与访问管理（IAM）

#### 3.2.1 SPIFFE/SPIRE 工作负载身份

**核心设计**：每个 Agent 实例拥有唯一的、可验证的工作负载身份（SVID）。

```
┌─────────────────────────────────────────────────────────────────┐
│              SPIFFE/SPIRE 身份生命周期                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 注册 (Registration)                                         │
│     ┌──────────┐         ┌──────────┐                          │
│     │  Agent   │ ──────▶ │ SPIRE    │                          │
│     │ 启动     │  提供   │ Server   │                          │
│     │          │ 节点证明 │          │                          │
│     └──────────┘         └────┬─────┘                          │
│                                │                                │
│  2. 身份颁发 (Issuance)                                        │
│                                ▼                                │
│                         ┌──────────┐                           │
│                         │  SVID    │                           │
│                         │ 颁发     │                           │
│                         │          │                           │
│                         │ spiffe://│                           │
│                         │quality-  │                           │
│                         │agent/    │                           │
│                         │test-gen- │                           │
│                         │01        │                           │
│                         └────┬─────┘                           │
│                                │                                │
│  3. 身份使用 (Usage)                                           │
│                                ▼                                │
│     ┌──────────┐         ┌──────────┐                          │
│     │  Agent   │◄───────►│  mTLS   │                          │
│     │ 通信     │  SVID   │ 握手    │                          │
│     └──────────┘         └──────────┘                          │
│                                                                 │
│  4. 轮换 (Rotation)                                            │
│     • SVID 自动轮换（默认 1 小时）                               │
│     • 旧 SVID 立即失效                                          │
│     • 无需人工干预                                              │
│                                                                 │
│  5. 撤销 (Revocation)                                          │
│     • Agent 异常行为 → 立即撤销 SVID                            │
│     • 被攻破节点 → 批量撤销 + 重新颁发                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**身份格式**：

```
SPIFFE ID: spiffe://quality-agent.mesh/{namespace}/{agent-type}/{instance-id}

示例：
- spiffe://quality-agent.mesh/prod/test-generator/agent-01
- spiffe://quality-agent.mesh/prod/execution-agent/agent-03
- spiffe://quality-agent.mesh/prod/raft-leader/node-01
```

#### 3.2.2 动态权限模型（ABAC + RBAC 混合）

| 模型 | 适用场景 | 示例 |
|------|---------|------|
| **RBAC** | 静态角色权限 | Test Generator Agent 默认拥有代码读取权限 |
| **ABAC** | 动态环境感知 | 同一 Agent 在非工作时间权限降级 |
| **Risk-Based** | 实时风险评估 | Agent 行为异常时，自动限制敏感操作 |

**权限决策流程**：

```python
# 伪代码：动态权限决策
def authorize_request(agent_id: str, action: str, resource: str, context: dict) -> Decision:
    # 1. 身份验证
    svid = verify_svid(agent_id)
    if not svid.valid:
        return Decision.DENY
    
    # 2. RBAC 基础检查
    role = get_role(svid)
    if not rbac_check(role, action, resource):
        return Decision.DENY
    
    # 3. ABAC 环境检查
    if not abac_check(context.time, context.location, context.device):
        return Decision.DENY
    
    # 4. 实时风险评估
    risk_score = risk_engine.evaluate(agent_id, action, context)
    if risk_score > THRESHOLD:
        # 触发增强验证（如多因素认证）
        return Decision.CHALLENGE
    
    # 5. 审计日志
    audit_logger.log(agent_id, action, resource, context, Decision.ALLOW)
    
    return Decision.ALLOW
```

### 3.3 安全通信层

#### 3.3.1 mTLS 全链路加密

| 通信场景 | 加密方式 | 证书管理 |
|---------|---------|---------|
| Agent ↔ Raft Leader | mTLS | SPIRE 自动颁发 |
| Agent ↔ Event Bus | mTLS | SPIRE 自动颁发 |
| Agent ↔ 共享记忆层 | mTLS + 字段级加密 | SPIRE + 应用层密钥 |
| Agent ↔ 外部工具 | mTLS（如支持）或 VPN | 手动配置 |

**mTLS 握手流程**：

```
Agent A                          Agent B
  │                                │
  │ ───── Client Hello ──────────▶ │
  │  + SVID (spiffe://.../agent-01)│
  │                                │
  │ ◀──── Server Hello ────────── │
  │  + SVID (spiffe://.../agent-02)│
  │  + 证书验证请求                │
  │                                │
  │ ───── 证书验证 ──────────────▶ │
  │  (通过 SPIRE 验证对端身份)      │
  │                                │
  │ ◀──── 加密通道建立 ────────── │
  │                                │
  │ ───── 加密数据传输 ──────────▶ │
  │                                │
```

#### 3.3.2 事件签名与防篡改

**核心机制**：每个事件必须携带发送者的数字签名，接收方验证签名和事件哈希。

```json
{
  "event_id": "evt-uuid-123",
  "event_type": "TaskCompleted",
  "timestamp": "2026-04-26T10:30:00Z",
  "source": {
    "agent_id": "agent-01",
    "spiffe_id": "spiffe://quality-agent.mesh/prod/test-generator/agent-01"
  },
  "payload": {
    "task_id": "task-456",
    "status": "success",
    "result_hash": "sha256:abc123..."
  },
  "signature": {
    "algorithm": "Ed25519",
    "value": "base64(signature_of_event_hash)"
  },
  "previous_event_hash": "sha256:prev456..."
}
```

**验证流程**：
1. 验证 `signature` 是否由 `spiffe_id` 对应的私钥签署
2. 验证 `payload` 的哈希是否与 `result_hash` 一致
3. 验证 `previous_event_hash` 是否与上一个事件的哈希匹配（哈希链）

### 3.4 Agent 执行环境安全

#### 3.4.1 gVisor 沙箱隔离

| 隔离维度 | 传统容器 | gVisor 沙箱 | 说明 |
|---------|---------|------------|------|
| **系统调用** | 直接访问内核 | 用户空间拦截 | 99% 系统调用不进入内核 |
| **文件系统** | 共享宿主机 | 独立虚拟文件系统 | Agent 只能看到允许的文件 |
| **网络** | 共享宿主机网络栈 | 独立网络栈 | 网络策略严格限制 |
| **资源** | cgroup 限制 | cgroup + 自定义限制 | 双重保障 |

**gVisor 配置示例**：

```yaml
# gvisor-sandbox-config.yaml
runtime: "runsc"
mode: "sandbox"  # 最严格模式

# 系统调用过滤
seccomp:
  default_action: "ENOSYS"  # 默认拒绝
  allowed_syscalls:
    - "read"
    - "write"
    - "openat"
    - "close"
    - "mmap"
    - "exit_group"
  # 禁止危险调用
  blocked_syscalls:
    - "execve"       # 禁止执行新程序
    - "ptrace"       # 禁止调试
    - "mount"        # 禁止挂载
    - "chroot"       # 禁止切换根目录

# 文件系统限制
filesystem:
  root: "/sandbox/root"
  read_only_paths:
    - "/app"
    - "/lib"
  writable_paths:
    - "/tmp"
    - "/var/log"
  max_size: "1GB"

# 网络限制
network:
  mode: "restricted"
  allowed_destinations:
    - "10.0.0.0/8"     # 内部网络
    - "172.16.0.0/12"  # 内部网络
  blocked_destinations:
    - "0.0.0.0/0"      # 禁止外网（除白名单）
  max_connections: 10

# 资源限制
resources:
  cpu: "2"
  memory: "4Gi"
  max_processes: 50
```

#### 3.4.2 运行时行为监控

**监控维度**：

| 维度 | 监控指标 | 异常行为示例 |
|------|---------|------------|
| **系统调用** | 调用频率、类型分布 | 短时间内大量 `openat`（可能是扫描文件系统） |
| **网络流量** | 连接数、流量大小、目标地址 | 连接到未知外部 IP（可能是数据外泄） |
| **文件访问** | 访问路径、读写频率 | 访问 `/etc/passwd`（可能是信息收集） |
| **资源使用** | CPU、内存、磁盘 I/O | 内存使用突增（可能是内存泄漏或攻击） |
| **API 调用** | 调用频率、参数模式 | 频繁调用敏感 API（可能是权限提升尝试） |

**异常检测与响应**：

```python
# 伪代码：运行时异常检测
def monitor_agent_behavior(agent_id: str):
    profile = load_behavior_profile(agent_id)
    
    while True:
        metrics = collect_runtime_metrics(agent_id)
        
        # 1. 规则匹配
        for rule in security_rules:
            if rule.match(metrics):
                trigger_alert(agent_id, rule)
                if rule.severity == "CRITICAL":
                    isolate_agent(agent_id)  # 立即隔离
        
        # 2. 异常检测模型
        anomaly_score = ml_model.predict(metrics, profile)
        if anomaly_score > 0.95:
            trigger_alert(agent_id, "ANOMALY_DETECTED")
            request_human_review(agent_id)
        
        # 3. 动态权限调整
        risk_score = calculate_risk_score(metrics)
        if risk_score > THRESHOLD:
            restrict_permissions(agent_id, "SENSITIVE_OPS")
        
        time.sleep(1)
```

## 五、安全度量与验证

### 5.1 安全指标体系

| 指标 | 目标值 | 测量方法 |
|------|-------|---------|
| **身份验证覆盖率** | 100% | 所有 Agent 通信是否都经过 mTLS |
| **异常检测准确率** | > 95% | 真阳性 / (真阳性 + 假阳性) |
| **异常检测召回率** | > 90% | 真阳性 / (真阳性 + 假阴性) |
| **隔离响应时间** | < 5 秒 | 从检测到隔离的时间 |
| **权限审计覆盖率** | 100% | 所有权限变更是否都有审计日志 |
| **SVID 轮换成功率** | > 99.9% | 轮换成功次数 / 总轮换次数 |
| **沙箱逃逸阻止率** | 100% | 成功阻止的逃逸尝试 / 总尝试 |

### 5.2 红队测试计划

| 测试场景 | 攻击向量 | 预期防御 |
|---------|---------|---------|
| **身份伪造** | 使用伪造的 SVID 连接 | SPIRE 验证失败，拒绝连接 |
| **中间人攻击** | 拦截 Agent 间通信 | mTLS 证书验证失败 |
| **权限提升** | 尝试访问超出权限的资源 | ABAC 策略拒绝，触发告警 |
| **沙箱逃逸** | 利用系统调用漏洞逃逸 | gVisor 拦截，记录日志 |
| **DDoS 攻击** | 大量伪造请求淹没系统 | 速率限制，异常检测触发 |
| **内部威胁** | 合法 Agent 被攻破后作恶 | 行为基线检测，自动隔离 |

## 八、风险评估与缓解

| 风险 | 可能性 | 影响 | 缓解措施 |

## 九、总结

零信任 Agent 网格不是对现有架构的颠覆，而是**安全增强**。它通过以下方式保护去中心化共识架构：

1. **身份层面**：每个 Agent 都有唯一、可验证、自动轮换的身份
2. **通信层面**：所有通信加密、签名、防篡改
3. **执行层面**：每个 Agent 在沙箱中运行，行为受监控
4. **权限层面**：动态、细粒度、基于风险的权限控制
5. **审计层面**：全链路可观测、可追溯

**核心价值**：即使某个 Agent 被攻破，攻击者也无法横向移动、无法篡改共识、无法窃取数据。
