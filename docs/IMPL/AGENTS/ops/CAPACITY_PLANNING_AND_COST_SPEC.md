## 1. 概述

### 1.1 文档目的

本规范定义了质量智能中台运维Agent的**容量规划和成本评估标准**，包括：
- 系统容量评估模型
- 资源需求计算
- 成本估算方法
- 容量规划流程

### 1.2 容量规划原则

| 原则 | 说明 |
|-----|------|
| **前瞻性** | 基于业务增长趋势预留扩展空间 |
| **成本效益** | 在满足SLO的前提下优化成本 |
| **弹性伸缩** | 支持按需扩缩容 |
| **可量化** | 所有规划基于数据驱动 |

### 2.2 容量评估公式

**理论容量公式：**
```
Capacity = min(
    Collector_capacity × Collector_replicas,
    Detector_capacity × Detector_replicas,
    Storage_capacity,
    Network_bandwidth_capacity
)
```

**实际可用容量：**
```
Usable_Capacity = Capacity × Safety_Factor × Degradation_Factor

其中：
- Safety_Factor = 0.7 (预留30%余量)
- Degradation_Factor = 0.85 (考虑性能波动)
```

## 3. 根因分析Agent容量规划

### 3.1 资源消耗模型

| 阶段 | CPU消耗 | 内存消耗 | 存储消耗 | 网络消耗 |
|-----|--------|--------|--------|--------|
| **数据收集** | 0.5核/请求 | 100MB/请求 | 5MB/请求 | 1MB/请求 |
| **关联分析** | 1核/请求 | 200MB/请求 | 10MB/请求 | 0.5MB/请求 |
| **根因推理** | 2核/请求 | 500MB/请求 | 2MB/请求 | 0.2MB/请求 |
| **报告生成** | 0.3核/请求 | 50MB/请求 | 50KB/请求 | 0.1MB/请求 |

**单次分析总资源消耗：**
```
CPU: 3.8核秒/请求
内存峰值: 850MB/请求
临时存储: 17.55MB/请求
网络流量: 1.8MB/请求
```

### 3.2 容量计算

**假设条件：**
- 日均故障数: 50
- 峰值故障数: 200/小时
- 平均分析时间: 2分钟
- SLA要求: 99.5%

**计算过程：**

```
Step 1: 计算并发分析能力需求

峰值并发 = 峰值故障数 × (平均分析时间 / 峰值窗口)
         = 200 × (2分钟 / 1小时)
         = 6.67 ≈ 7个并发分析

Step 2: 计算所需副本数

单副本并发能力 = 3个并发/副本 (考虑到资源竞争)
所需副本数 = ceil(7 / 3) = 3副本

Step 3: 计算资源需求

CPU = 3副本 × 4核/副本 = 12核
内存 = 3副本 × 8GB/副本 = 24GB
```

### 3.3 推荐配置

| 规模 | 日故障数 | 推荐副本数 | CPU | 内存 | 存储 |
|-----|---------|-----------|-----|------|-----|
| **小规模** | <10 | 2 | 4核 | 8GB | 100GB |
| **中规模** | 10-50 | 3 | 12核 | 24GB | 200GB |
| **大规模** | 50-200 | 5 | 20核 | 40GB | 500GB |
| **超大规模** | >200 | 10+ | 40核+ | 80GB+ | 1TB+ |

### 3.4 LLM资源需求

| LLM模型 | 单次推理CPU | 单次推理内存 | 单次推理GPU | 单次推理时间 | QPS能力 |
|--------|-----------|------------|------------|------------|---------|

## 4. 异常检测Agent容量规划

### 4.1 资源消耗模型

| 阶段 | CPU消耗 | 内存消耗 | 存储消耗 | 网络消耗 |
|-----|--------|--------|--------|--------|
| **数据采集** | 0.1核/指标 | 10MB/1000指标 | 1KB/指标 | 0.5KB/指标 |
| **时序检测** | 0.5核/指标 | 100MB/10000指标 | 5KB/指标 | 0 |
| **统计检测** | 0.2核/指标 | 50MB/10000指标 | 1KB/指标 | 0 |
| **模式检测** | 1核/1000指标 | 200MB/1000指标 | 0 | 0 |

**单指标检测资源消耗：**
```
CPU: 0.8核秒/指标
内存峰值: 360KB/指标
临时存储: 7KB/指标
```

### 4.2 容量计算

**假设条件：**
- 监控指标数: 100,000
- 采集频率: 10秒/次
- 检测算法: 时序+统计+模式融合
- SLA要求: 99.9%

**计算过程：**

```
Step 1: 计算采集QPS

采集QPS = 100,000指标 / 10秒 = 10,000 QPS

Step 2: 计算检测QPS

检测QPS = 采集QPS = 10,000 QPS

Step 3: 计算所需Collector副本数

单个Collector处理能力 = 5,000 QPS
所需Collector副本 = ceil(10,000 / 5,000) = 2副本

Step 4: 计算所需Detector副本数

单个Detector处理能力 = 2,000 QPS
所需Detector副本 = ceil(10,000 / 2,000) = 5副本

Step 5: 计算GPU需求 (模式检测)

GPU需求 = 10,000指标 / 1000指标/GPU = 10 GPU
考虑冗余 = 12 GPU (2台8卡服务器)
```

### 4.3 推荐配置

| 规模 | 指标数量 | Collector副本 | Detector副本 | CPU | 内存 | GPU |
|-----|---------|-------------|------------|-----|------|-----|
| **小规模** | <10,000 | 1 | 2 | 4核 | 8GB | 0 |
| **中规模** | 10,000-50,000 | 2 | 3 | 16核 | 32GB | 2×A100 |
| **大规模** | 50,000-200,000 | 3 | 5 | 32核 | 64GB | 4×A100 |

## 5. 存储容量规划

### 5.1 存储需求估算

**数据保留策略：**

| 数据类型 | 热层保留 | 温层保留 | 冷层保留 | 总保留 |
|--------|--------|--------|--------|-------|
| 原始指标 | 7天 | 30天 | 90天 | 127天 |
| 告警事件 | 30天 | 60天 | 180天 | 270天 |
| 分析报告 | 30天 | 90天 | 365天 | 485天 |
| 影子对比数据 | 30天 | 90天 | 365天 | 485天 |
| 日志 | 7天 | 30天 | 90天 | 127天 |

**存储容量计算：**

```
月存储量 = Sum(每日数据量 × 保留天数 × 压缩率)

原始指标月存储：
= 100,000指标 × 8640次/天 × 30天 × 1KB × 0.3压缩率
= 77.76 GB/月

告警事件月存储：
= 1,000事件/天 × 30天 × 10KB × 0.4压缩率
= 0.12 GB/月

分析报告月存储：
= 50报告/天 × 30天 × 100KB × 0.5压缩率
= 0.75 GB/月

影子对比数据月存储：
= 50对比/天 × 30天 × 500KB × 0.4压缩率
= 3 GB/月

总计：约 82 GB/月
```

### 5.2 存储推荐配置

| 存储类型 | 介质 | 容量规划 | 适用场景 |
|---------|-----|---------|---------|
| **热层** | SSD NVMe | 500GB-1TB | PostgreSQL (JSONB) |
| **温层** | SSD | 2-5TB | ClickHouse |
| **冷层** | 对象存储 | 10-50TB | S3/OBS Parquet |

### 6.2 成本优化建议

| 优化项 | 优化方法 | 节省比例 |
|-------|---------|---------|
| **实例规格** | 使用抢占式实例 | 60-70% |
| **存储层级** | 冷热数据分离 | 40-50% |
| **计算资源** | GPU共享/分时复用 | 30-40% |
| **网络** | 区域就近部署 | 20-30% |
| **预留实例** | 预留1年合约 | 30-40% |

### 6.3 成本估算公式

```
月成本 = C_compute + C_storage + C_network + C_misc

其中：
C_compute = Sum(实例规格单价 × 数量 × 使用率) × 优化系数
C_storage = Sum(存储容量 × 单价) × 访问频率系数
C_network = 流量费用 + 带宽费用
C_misc = 监控、日志等其他服务费用

年成本 = 月成本 × 12 × (1 - 年度折扣)
```

### 10.2 容量计算器

```python
class CapacityCalculator:
    """容量计算器"""

    def calculate_rca_capacity(self, daily_faults: int, peak_factor: float = 4.0) -> dict:
        """计算RCA容量需求"""
        avg_faults_per_hour = daily_faults / 24
        peak_faults_per_hour = avg_faults_per_hour * peak_factor
        peak_concurrent = peak_faults_per_hour * (2 / 60)  # 2分钟分析时间

        replicas = max(2, int(peak_concurrent / 3))
        cpu = replicas * 4
        memory = replicas * 8
        storage = daily_faults * 30 * 100 / 1024  # GB

        return {
            "replicas": replicas,
            "cpu_cores": cpu,
            "memory_gb": memory,
            "storage_gb": storage,
            "estimated_cost": replicas * 800  # 元/月
        }

    def calculate_anomaly_capacity(self, metric_count: int, interval_seconds: int = 10) -> dict:
        """计算异常检测容量需求"""
        qps = metric_count / interval_seconds

        collector_replicas = max(1, int(qps / 5000))
        detector_replicas = max(2, int(qps / 2000))
        gpu_count = max(0, int(metric_count / 1000 / 4))  # 每4 GPU处理1000指标

        cpu = collector_replicas * 4 + detector_replicas * 4
        memory = collector_replicas * 8 + detector_replicas * 32
        storage = metric_count * 8640 * 30 / 1024 / 1024 * 1  # GB

        return {
            "collector_replicas": collector_replicas,
            "detector_replicas": detector_replicas,
            "gpu_count": gpu_count,
            "cpu_cores": cpu,
            "memory_gb": memory,
            "storage_gb": storage
        }
```
