## 1. 概述

### 1.1 Agent定位

根因分析Agent是质量智能中台运维场景的核心Agent之一，负责在故障发生时自动推理故障根因，为运维团队提供问题定位和修复建议服务。

### 1.2 核心价值

| 价值维度 | 描述 |
|---------|------|
| **缩短MTTR** | 智能根因分析，快速定位问题，目标MTTR缩短 > 40% |
| **知识沉淀** | 故障案例自动沉淀，避免重复踩坑 |
| **经验复用** | 历史故障经验转化为可复用知识 |

### 1.3 与其他Agent的关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    根因分析Agent在价值链中的位置                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  异常检测 ──▶ 告警收敛 ──▶ 【根因分析】 ──▶ 修复执行          │
│                              │                                   │
│                              ▼                                   │
│                       输出根因报告                               │
│                       提供修复建议                               │
│                       触发知识沉淀                               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 3. 具体功能点详细描述

### 3.1 数据收集层

#### G1.1 告警数据收集

| 属性 | 规格说明 |
|-----|---------|
| **数据来源** | 监控系统、异常检测Agent输出 |
| **收集范围** | 故障发生前后2小时的告警事件 |
| **数据格式** | JSON，含alert_id、name、severity、start_time、end_time |
| **关联字段** | alert_id、service_name、host |

#### G1.2 指标数据收集

| 属性 | 规格说明 |
|-----|---------|
| **数据来源** | Prometheus/InfluxDB |
| **收集范围** | 故障发生时刻的指标快照 |
| **采样频率** | 1分钟/次（故障期间5秒/次） |
| **指标类型** | CPU、内存、磁盘、网络、应用QPS/延迟/错误率 |

#### G1.3 变更数据收集

| 属性 | 规格说明 |
|-----|---------|
| **变更类型** | 代码发布、配置变更、架构变更、流量变更 |
| **时间窗口** | 故障发生前2小时 + 发生后30分钟 |
| **数据源** | 发布平台、配置中心、Nginx配置、负载均衡器 |
| **关联方法** | 时间线对齐 + 拓扑影响分析 |

#### G1.4 拓扑数据收集

| 属性 | 规格说明 |
|-----|---------|
| **数据来源** | 服务注册中心(Consul/Nacos)、CMDB |
| **拓扑结构** | 服务→组件→实例→指标 |
| **依赖关系** | 调用依赖、访问依赖、数据依赖 |
| **更新频率** | 实时同步 |

#### G1.5 知识库查询

| 属性 | 规格说明 |
|-----|---------|
| **知识库类型** | 历史故障案例、运维手册、最佳实践、专家经验 |
| **查询方式** | 语义相似度 + 关键词匹配 + 标签筛选 |
| **返回数量** | Top-5最相关案例 |
| **知识更新** | 故障处理完成后自动沉淀新案例 |

#### G1.6 数据质量校验

| 属性 | 规格说明 |
|-----|---------|
| **校验类型** | 完整性、时效性、一致性、可用性 |
| **完整性检查** | 告警数量、指标覆盖率、日志覆盖率 |
| **时效性检查** | 数据延迟阈值：告警<30s、指标<60s、日志<120s |
| **一致性检查** | 多数据源交叉验证（如同一告警在监控系统和ITS中状态一致） |
| **可用性检查** | 数据源健康状态探测，自动屏蔽不可用数据源 |
| **降级触发** | 数据缺失率>20%或关键数据源不可用时触发降级 |
| **异常处理** | 标记可疑数据但不删除，保留原始数据供人工核查 |

### 3.2 关联分析层

#### G2.1 时序关联分析

| 属性 | 规格说明 |
|-----|---------|
| **算法** | Granger因果检验 + 时序卷积网络 |
| **时间窗口** | 告警前后5分钟 |
| **最小置信度** | 0.8 |
| **输出** | 告警因果关系图 |
| **示例** | 告警A发生在告警B之前，且A能预测B，则A→B |

#### G2.2 拓扑关联分析

| 属性 | 规格说明 |
|-----|---------|
| **分析范围** | 故障服务及其上下游依赖服务 |
| **影响路径** | 从故障服务到用户侧的影响链路 |
| **传播模型** | 告警传播时间模型（每跳 <= 30秒） |

#### G2.3 变更关联分析

```
变更关联度计算:
├── 时间接近度 (40%)
│   └── 变更时间距离故障时间越近，关联度越高
│
├── 拓扑影响度 (30%)
│   └── 变更的组件是否在故障影响路径上
│
├── 历史故障率 (30%)
│   └── 该组件历史变更引发故障的概率
│
└── 综合关联度 = 0.4×时间接近度 + 0.3×拓扑影响度 + 0.3×历史故障率
```

#### G2.4 日志关联分析

| 属性 | 规格说明 |
|-----|---------|
| **日志范围** | 故障发生前后5分钟内的ERROR/WARN日志 |
| **聚类方法** | 基于日志模板的语义聚类 |
| **关联规则** | 日志时间线与告警时间线对齐 |

### 3.3 根因推理层

#### G3.1 知识图谱推理

| 属性 | 规格说明 |
|-----|---------|
| **图谱结构** | 实体: 服务/组件/指标/告警; 关系: 依赖/调用/监控 |
| **推理算法** | 基于图神经网络的推理 (GNN) |
| **推理深度** | 最多5层依赖追溯 |
| **知识更新** | 故障复盘后自动更新图谱 |
| **覆盖范围** | 全量核心服务覆盖 |

#### G3.2 因果推断

| 属性 | 规格说明 |
|-----|---------|
| **算法** | PC算法 + Do-calculus |
| **因果图构建** | 基于时序数据和拓扑数据构建 |
| **根因判定** | 基于介入分析的因果效应计算 |

#### G3.3 LLM语义推理

| 属性 | 规格说明 |
|-----|---------|
| **模型** | Qwen/Qwen-72B 或同等能力LLM |
| **Prompt模板** | 预定义根因分析Prompt，含拓扑/告警/日志上下文 |
| **推理超时** | 30秒 |
| **输出格式** | 结构化JSON: {root_cause, confidence, reasoning_path} |
| **安全约束** | 不执行任何写操作，仅分析推理 |

#### G3.4 多推理融合

```
融合策略:
├── 投票机制
│   ├── 3个推理器结果一致 → 高置信度
│   ├── 2个推理器结果一致 → 中置信度
│   └── 3个结果都不一致 → 低置信度，需要人工介入
│
├── 置信度加权
│   ├── 知识图谱权重: 0.35
│   ├── 因果推断权重: 0.35
│   └── LLM推理权重: 0.30
│
└── 冲突解决
    └── 当结果冲突时，优先选择置信度最高的
        若置信度相同，优先选择知识图谱推理结果
```

#### G3.5 置信度评估

| 置信度等级 | 分数范围 | 说明 |
|-----------|---------|------|
| **高** | >= 0.85 | 根因可信，可直接建议处理 |
| **中** | 0.60 - 0.85 | 根因待验证，建议优先排查 |
| **低** | < 0.60 | 需要人工介入分析 |

#### G3.6 降级策略与异常处理

| 场景 | 降级策略 | 处理机制 |
|-----|---------|---------|
| **LLM服务不可用** | 自动切换到知识图谱+因果推断双推理模式 | 降级置信度×0.9，标记`llm_skipped=true` |
| **LLM超时（>30s）** | 跳过LLM推理，使用双推理结果 | 超时计数+1，超过3次触发告警 |
| **知识图谱不可用** | 自动切换到因果推断+LLM双推理模式 | 置信度阈值提升至0.75，标记`kg_skipped=true` |
| **因果推断失败** | 自动切换到知识图谱+LLM双推理模式 | 标记`causal_skipped=true` |
| **单一推理器失败** | 使用剩余两个推理器加权融合 | 重新计算权重（失败推理器权重置零） |
| **两个推理器同时失败** | 降级到基于规则的简化分析模式 | 输出简化报告，置信度≤0.5，标记`degraded_mode=true` |
| **知识库查询失败** | 使用默认案例模板 | 降级通知推送给运维人员 |
| **数据质量不达标** | 降低关联分析精度要求（置信度阈值降至0.6） | 在报告中标记`data_quality_warning=true` |

**降级决策树：**
```
开始分析
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 1: 检查各推理器可用性                                 │
│  ├── LLM可用？                                              │
│  ├── 知识图谱可用？                                         │
│  └── 因果推断可用？                                         │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 2: 决策降级策略                                       │
│                                                             │
│  3个推理器全可用 → 三推理融合（标准模式）                    │
│  2个推理器可用 → 双推理融合（降级模式A）                     │
│  1个推理器可用 → 单推理输出（降级模式B）                     │
│  0个推理器可用 → 规则引擎兜底（紧急模式）                    │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Step 3: 记录降级元数据                                      │
│  - degraded: bool                                           │
│  - degradation_reason: string                              │
│  - active_inferencers: string[]                            │
│  - fallback_triggered: string[]                             │
└─────────────────────────────────────────────────────────────┘
```

**异常处理规范：**
- 所有异常必须记录到`processing_details.errors`数组中
- 降级触发后自动发送告警通知（P1级）
- 降级分析结果需在报告中明确标注，供人工复审
- 降级事件纳入运维度量指标统计

## 5. 可量化性能指标

| 指标类别 | 指标名称 | 目标值 | 测量方法 |
|---------|---------|--------|---------|
| **准确性** | 根因定位准确率 | >= 85% | 人工复盘验证/总分析数 |
| **准确性** | Top-3根因命中率 | >= 95% | 真实根因在候选列表位置 |

```protobuf
syntax = "proto3";

package root_cause_analysis;

service RootCauseAnalysisService {
  rpc TriggerAnalysis (AnalysisRequest) returns (AnalysisResponse);
  rpc BatchTriggerAnalysis (BatchAnalysisRequest) returns (BatchAnalysisResponse);
  rpc GetAnalysisHistory (HistoryRequest) returns (HistoryResponse);
  rpc GetAnalysisDetail (DetailRequest) returns (RCADetailResponse);
  rpc SubscribeAnalysis (stream AnalysisControl) returns (stream AnalysisProgress);

}

message AnalysisRequest {
  string fault_id = 1;
  repeated string alert_ids = 2;

  ContextWindow context = 5;

enum AnalysisMode {
```

### 8.1 影子模式要点

| 配置项 | 规格 |
|-------|------|

| **对比基准** | 人工根因分析结果 |

### 8.2 影子模式特殊要求

| 要求 | 说明 |
|-----|------|

| **复盘对比** | 重大故障必须做AI vs 人工对比复盘 |
| **案例积累** | 积累不少于50个真实故障案例 |
